🎬 Antes de Empezar

Si vienes de trabajar con GitLab, lo principal que debes saber es que el codigo ahora vive en GitHub. En lugar de entrar a gitlab.com/..., ahora entras a github.com/GoEcosystemDH. Los conceptos son los mismos (repositorios, ramas, merge requests), pero algunos nombres cambian. Por ejemplo, lo que en GitLab se llama "Merge Request", en GitHub se llama "Pull Request" (PR). Hacen exactamente lo mismo.

Dato clave Nuestro codigo fuente, Pull Requests y pipelines de CI/CD ahora estan en github.com/GoEcosystemDH. La planificacion (sprints, tareas) sigue en Azure DevOps. La documentacion tecnica esta en la Wiki.

Como clonar un repositorio desde GitHub

"Clonar" significa descargar una copia del codigo a tu computadora para poder trabajar en el. Si antes escribias git clone git@gitlab.com/..., ahora solo cambia la direccion:

# Ejemplo: clonar el proyecto "costos-api" a tu computadora git clone git@github.com:GoEcosystemDH/costos-api.git # Entrar a la carpeta del proyecto cd costos-api # Verificar que estas conectado al repositorio correcto git remote -v # origin git@github.com:GoEcosystemDH/costos-api.git (fetch) # origin git@github.com:GoEcosystemDH/costos-api.git (push)
Primer paso Antes de clonar, asegurate de tener tu entorno configurado: SSH keys, Git, y acceso a la organizacion. La guia completa esta en la wiki.

🔄 Tu Dia a Dia en 6 Pasos

Cada vez que necesites agregar algo nuevo, corregir un error o hacer cualquier cambio en el codigo, vas a seguir estos 6 pasos. Siempre. No importa si el cambio es grande o chico.

Paso 1
Actualizar
Bajar lo mas reciente
→
Paso 2
Crear rama
Tu espacio de trabajo
→
Paso 3
Programar
Hacer cambios y guardar
→
Paso 4
Subir y pedir revision
Push + Pull Request
→
Paso 5
Revision
Un companero revisa
→
Paso 6
Integrar
Merge + deploy auto
  1. Actualizar tu copia local. Antes de empezar, asegurate de tener la version mas reciente del codigo. Esto evita conflictos con los cambios de otros.
  2. Crear una rama (branch). Piensa en una rama como tu mesa de trabajo personal. Trabajas ahi sin afectar a nadie. Cuando termines, pediras que integren tus cambios.
  3. Programar y guardar tus avances (commits). Haces tus cambios en el codigo y los vas "guardando" con mensajes descriptivos. Cada guardado se llama un commit.
  4. Subir tu trabajo y pedir revision (Push + Pull Request). "Push" sube tu rama a GitHub. Luego creas un "Pull Request" (PR): es como decir "ya termine, revisen por favor".
  5. Revision por un companero (Code Review). Al menos una persona del equipo revisa tu codigo. Te puede pedir cambios o aprobarlo.
  6. Integrar (Merge). Cuando tu PR es aprobado, se integra al codigo principal. Un robot automatico se encarga de compilar, probar y desplegar.

Ejemplo real: Maria necesita agregar un boton de exportar PDF en el modulo de costos

Maria recibio un ticket (COST-456) para agregar la funcionalidad de exportar reportes a PDF. Estos son los comandos exactos que ejecutaria, desde el inicio hasta el final:

# PASO 1: Actualizar su copia local git checkout develop git pull origin develop # PASO 2: Crear su rama de trabajo git checkout -b feature/COST-456-export-pdf # PASO 3: Maria programa la funcionalidad... # (edita archivos, agrega el boton, la logica de PDF, etc.) # Cuando tiene un avance significativo, guarda (commit): git add src/components/ExportButton.tsx git commit -m "feat(costos): add PDF export button to report view" # Maria sigue trabajando y hace otro commit: git add src/services/pdfGenerator.ts src/utils/formatReport.ts git commit -m "feat(costos): implement PDF generation service" # PASO 4: Subir la rama a GitHub y crear Pull Request git push -u origin feature/COST-456-export-pdf # Luego va a GitHub y crea el Pull Request desde la interfaz web # (o usa el link que Git le muestra en la terminal) # PASO 5: Su companero Carlos revisa el PR y lo aprueba # PASO 6: Maria (o Carlos) hacen click en "Merge" en GitHub # El pipeline automatico compila, prueba y despliega # Despues del merge, Maria limpia su rama local: git checkout develop git pull origin develop git branch -d feature/COST-456-export-pdf

🌿 Las Ramas (Branches) — ¿Por que no puedo subir directo a main?

Imaginate un restaurante. main es el restaurante abierto al publico: todo lo que esta ahi debe funcionar perfectamente porque los clientes lo estan usando. develop es la cocina, donde los chefs prueban platos nuevos antes de servirlos. Y tu feature es tu mesa de trabajo personal en la cocina, donde preparas tu receta sin molestar a los demas.

No puedes subir directo a main porque seria como servir un plato a medio preparar a un cliente. Primero lo pruebas en tu mesa (feature), despues pasa a la cocina general (develop), y finalmente, cuando esta listo, se sirve al publico (main).

main El restaurante abierto (produccion). Solo recibe codigo ya probado y aprobado, via release/* o hotfix/*. Bloqueada
develop La cocina (integracion). Aqui se juntan todas las funcionalidades terminadas antes de ir a produccion. PR Requerido
feature/* Tu mesa de trabajo. Creas una para cada tarea. Se crea desde develop y vuelve a develop. Libre
release/* El plato listo para servir. Prepara una version para produccion. Va a main y a develop. PR Requerido
hotfix/* Emergencia en produccion. Correccion urgente. Se crea desde main y va a main y develop. PR Requerido
Rama Para que sirve Quien puede escribir Ejemplo de nombre
main Codigo en produccion (lo que usan los usuarios) Nadie directamente — solo via merge main
develop Integracion de funcionalidades terminadas Solo via Pull Request aprobado develop
feature/* Desarrollar una funcionalidad nueva El desarrollador asignado feature/COST-456-export-pdf
release/* Preparar una version para produccion Lider tecnico / DevOps release/2.4.0
hotfix/* Corregir un error urgente en produccion El desarrollador asignado + aprobacion rapida hotfix/fix-login-crash

📝 Mensajes de Commit — El formato obligatorio

Un commit es como "guardar tu trabajo" con una descripcion de lo que hiciste. Piensa en ello como el historial de cambios: si algo se rompe, podemos ver exactamente que se cambio y por que.

Para que todos entendamos los mensajes de la misma forma, usamos un formato estandar llamado Conventional Commits. Es simple: escribes el tipo de cambio, el modulo donde lo hiciste, y una descripcion corta.

El formato

tipo(modulo): descripcion breve de lo que hiciste

tipo = que clase de cambio es  |  modulo = en que parte del proyecto  |  descripcion = que hiciste, en pocas palabras

Tipos de commit con ejemplos reales

Tipo Significado Ejemplo real
feat Funcionalidad nueva feat(costos): add PDF export to report view
fix Correccion de un error fix(his): resolve null pointer in patient search
docs Cambios en documentacion docs(rcf): update API endpoint documentation
refactor Reorganizar codigo sin cambiar funcionalidad refactor(citapp): extract date validation to utility
test Agregar o modificar pruebas test(costos): add unit tests for invoice calculator
chore Tareas de mantenimiento chore(his): upgrade Spring Boot to 3.2
ci Cambios en el pipeline automatico ci(rcf): add SonarQube step to pipeline
perf Mejora de rendimiento perf(citapp): optimize appointment query with index

Mal vs. Bien

MAL
git commit -m "cambios" git commit -m "fix bug" git commit -m "wip" git commit -m "update file"
BIEN
git commit -m "feat(costos): add PDF export button" git commit -m "fix(his): resolve timeout on patient search" git commit -m "refactor(rcf): simplify claim validation logic" git commit -m "test(citapp): add tests for appointment flow"
Importante Si tu mensaje no sigue el formato, el pipeline automatico lo va a rechazar. No es opcional — es un check automatico que valida cada commit.

🔍 Pull Requests — Pedir revision antes de integrar

Un Pull Request (PR) es como decir: "Ya termine mi trabajo. Por favor, revisen antes de publicar." Es obligatorio: nadie puede subir codigo directamente sin que al menos otra persona lo revise. Esto nos protege de errores, mejora la calidad del codigo y ayuda a compartir conocimiento en el equipo.

El proceso paso a paso

1
Creas el Pull Request
Desde tu rama feature hacia develop. Llenas el titulo y la descripcion usando el template.
2
Checks automaticos
Un robot verifica tu codigo: que compile, pase las pruebas, cumpla el formato de commits, y no tenga errores de estilo.
3
Code Review por un companero
Al menos una persona revisa tu codigo linea por linea. Puede aprobar, pedir cambios, o hacer preguntas.
4
Merge
Cuando todo esta aprobado y los checks pasan, se integra tu codigo a develop. El pipeline se encarga del resto.

Ejemplo de un buen PR

feat(costos): add PDF export to report view
¿Que hace este PR?
Agrega un boton "Exportar PDF" en la vista de reportes del modulo de costos. Al hacer click, genera un PDF con el reporte actual usando la libreria jsPDF.

¿Como probarlo?
1. Ir a Costos > Reportes > Seleccionar cualquier reporte
2. Click en "Exportar PDF"
3. Verificar que el PDF descargado contiene los datos del reporte

Ticket: COST-456

Tipo de cambio Aprobaciones necesarias Quien aprueba
Funcionalidad normal, bug fix, docs 1 aprobacion Cualquier miembro del equipo
Cambio de arquitectura, modelos de datos, APIs publicas 2 aprobaciones Al menos 1 lider tecnico o arquitecto
Hotfix (emergencia en produccion) 1 aprobacion Lider tecnico o DevOps

🚀 El Pipeline — ¿Que pasa cuando hago merge?

Cuando tu codigo se aprueba y se integra (merge), no tienes que hacer nada mas. Un proceso automatico (llamado "pipeline") se encarga de todo lo demas: compilar tu codigo, ejecutar las pruebas, y desplegarlo. Piensa en ello como una linea de ensamblaje en una fabrica: tu entregas la pieza terminada y la maquina se encarga de armarla, revisarla y empaquetarla.

Build
Compilar el codigo
→
Test
Ejecutar pruebas automaticas
→
Staging
Desplegar en ambiente de pruebas
→
Aprobacion
Alguien autoriza el pase
→
Produccion
En vivo para los usuarios

¿Que pasa en cada etapa?

Etapa ¿Que hace? ¿Necesitas hacer algo?
Build Compila tu codigo y genera el paquete (como convertir ingredientes en un plato armado). Tambien crea la imagen Docker si aplica. No. Es automatico.
Test Ejecuta todas las pruebas automaticas (unitarias, de estilo, de seguridad). Si algo falla, te avisa. Solo si falla: corrige y vuelve a subir.
Staging Despliega tu codigo en un ambiente de pruebas identico a produccion. Sirve para probar antes de que llegue a los usuarios reales. No. Es automatico.
Aprobacion Un lider tecnico o el equipo de QA verifica que todo se ve bien en staging antes de pasar a produccion. Solo si eres el aprobador.
Produccion Tu codigo llega a los usuarios reales. El despliegue se hace sin interrumpir el servicio (zero-downtime). No. Es automatico.
Si algo falla Cuando una etapa falla, recibiras una notificacion. Ve al PR en GitHub, revisa los logs del check que fallo, corrige tu codigo, y sube un nuevo commit. El pipeline se vuelve a ejecutar automaticamente.

⚡ Referencia Rapida de Comandos

Estos son los comandos de Git que usaras con mas frecuencia. Estan organizados por el momento en el que los necesitas. Si no recuerdas alguno, vuelve aqui.

Empezar a trabajar

Clonar un repo (primera vez) git clone git@github.com:GoEcosystemDH/mi-repo.git
Ir a la rama develop git checkout develop
Bajar lo mas reciente git pull origin develop
Crear tu rama de trabajo git checkout -b feature/TICKET-123-mi-cambio

Guardar mi trabajo

Ver que archivos cambiaste git status
Ver los cambios en detalle git diff
Agregar archivos al commit git add src/mi-archivo.ts
Guardar con mensaje git commit -m "feat(modulo): descripcion del cambio"

Pedir revision

Subir tu rama a GitHub git push -u origin feature/TICKET-123-mi-cambio
Crear PR desde terminal (opcional) gh pr create --base develop --title "feat(modulo): descripcion"

Despues del merge

Volver a develop git checkout develop
Actualizar develop git pull origin develop
Borrar tu rama local (ya no la necesitas) git branch -d feature/TICKET-123-mi-cambio

Si tu rama se desactualizo

Traer los cambios nuevos a tu rama git checkout feature/TICKET-123-mi-cambio && git merge develop
